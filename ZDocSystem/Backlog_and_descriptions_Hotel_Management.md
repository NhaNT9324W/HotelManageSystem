### 5. Bảng tổng hợp các Business Rules (toàn bộ)

| **ID** | **Business Rule** | **Business Rule Description** |
| --- | --- | --- |
| BR-01 | Password Encoding | User’s password must be encoded with MD5 hashing. |
| BR-02 | Email uniqueness | Email address must be unique within the system. |
| BR-03 | Login credential verification | User must provide valid username and password. |
| BR-04 | Account lockout policy | After 5 consecutive failed login attempts, account is locked for 15 minutes. |
| BR-05 | Session termination | All session tokens must be invalidated. |
| BR-06 | Cache clearing enforcement | Pressing the Back button to review old data is not allowed. |
| BR-07 | Password Policy | The new password must satisfy the system password policy, including minimum length, uppercase letters, lowercase letters, numbers, and special characters. |
| BR-08 | Account management authorization | Only Admin users can create, edit, or delete accounts. |
| BR-09 | Account activity auditing | All account management actions must be recorded in the audit log. |
| BR-10 | Email immutability | Email cannot be changed after account creation. |
| BR-11 | Active Admin protection | Admin cannot delete their own currently logged-in account. |
| BR-12 | Account privacy protection | Sensitive information such as passwords should absolutely not be returned in the API list. Information such as phone numbers and citizen identification numbers must be enclosed in asterisks (*). |
| BR-13 | Data display | Account information must be displayed in a table format. |
| BR-14 | Permission-based data scoping | Restrict data visibility by role: Hotel Manage see hotel-specific users; Admin see all system users. |
| BR-15 | Hotel Info Authorization | Only Admin users can edit the global hotel information. |
| BR-16 | System Config Auditing | All changes made to the hotel's core configuration must be recorded in the audit log for traceability. |
| BR-17 | Mandatory Hotel Fields | Hotel Name, Main Contact Number, and Physical Address cannot be left blank during an update. |
| BR-18 | Room Data Visibility | Admin users have full CRUD access to all room configurations and categories. |
| BR-19 | Room Number Uniqueness | Every room within the hotel must have a unique identifier (Room Number). The system must reject duplicate room numbers. |
| BR-20 | Logical Capacity & Pricing | Room Base Price and Max Capacity must be numeric values strictly greater than 0. |
| BR-21 | Initial Room Status | Newly created rooms can only be set to 'Available' or 'Maintenance' by default; they cannot be set to 'Occupied' or 'Reserved' during creation. |
| BR-22 | Guest data entry | Full name and at least one contact method (phone or email) are required to create a guest profile. |
| BR-23 | Unique guest identification | Each guest is identified by a unique system-generated Guest ID. Duplicate names or phones are allowed but flagged for manual check. |
| BR-24 | Profile modification permission | Only receptionists can create or edit guest profiles. Admin and Hotel Manager have read-only access. |
| BR-25 | Auto-generated Guest ID | Each guest profile is assigned a unique, system-generated Guest ID. The receptionist cannot change it. |
| BR-26 | Duplicate contact warning | If a phone number or email already exists in another guest profile, the system shows a warning but still allows creation (because two guests could share a phone number, e.g., family members). |
| BR-27 | (Không có) | – |
| BR-28 | Edit permission | Only receptionists can edit guest profiles. Admin and Hotel Manager cannot edit (view only). |
| BR-29 | History visibility | All reservations linked to the same guest ID are visible to receptionists, regardless of status (completed, cancelled, no-show). |
| BR-30 | No modification in history | The guest history screen is read-only. To modify a reservation, receptionist must go to the reservation management module. |
| BR-31 | Service Management Authorization | Only Admin and Hotel Manager roles are permitted to manage hotel services. |
| BR-32 | Service Activity Auditing | All create, edit, and delete actions on hotel services must be recorded in the audit log. |
| BR-33 | Data Display Format | Service information must be displayed in a table format with pagination. |
| BR-34 | Service Name Uniqueness | Service name must be unique within the hotel system. |
| BR-35 | Service Price Validation | Service price must be a positive numeric value greater than 0. |
| BR-36 | Search Result Display | Search results must support partial matching and be displayed in paginated format. |
| BR-37 | Active Service Protection | A service that is currently linked to active reservations cannot be deleted. |
